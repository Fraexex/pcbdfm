# 产品经理用 Claude 从零做 PCB DFM 工具——从痛点到原型

> 这是「AI 实战：用 Claude 从零做 PCB DFM 工具」专栏的第一篇。完整记录一个产品经理如何用 AI 编程助手，从零构建一个纯浏览器端的 PCB 可制造性分析工具。

---

## 一、痛点：硬件工程师投板前的焦虑

做过硬件的人都知道，PCB 设计完成后、发给板厂之前，最怕什么？

**DFM 不过关。**

走线太细、间距太小、钻孔不够大、丝印盖住了焊盘……任何一个问题都意味着打样返工。一次返工就是一周起步，如果是多层板，时间成本更不用说。

那现有的 DFM 检查工具怎么样呢？

- **CAM350 / DFM Chef**：专业，但贵，而且要本地安装，学习成本高
- **板厂自带的在线检查**：大部分只做基础校验，检查规则不透明，结果不可控
- **开源工具**：生态分散，大多需要命令行操作，对非开发人员不友好

所以我就想：能不能做一个**免费、浏览器打开就能用、检查规则透明可调**的 DFM 工具？

但我遇到一个现实问题——我是个产品经理，不是程序员。

传统路径下，这个想法基本就止步了。但现在有了 Claude，事情变得不一样。

---

## 二、挑战评估：这件事有多难

在动手之前，我先梳理了一下技术难点：

**1. Gerber RS-274X 格式解析**

Gerber 是 PCB 行业的标准文件格式，类似于 PCB 的 "PDF"。但它不是普通文本，有自己的指令体系——光圈定义、画线指令、闪点、区域填充，每种都有不同的语法规则。

**2. 纯前端约束**

我的目标是做一个零服务器的工具——打开 HTML 就能用。这意味着不能用后端解析，也不能装 npm 包引入现成的解析库。

**3. DFM 检查需要空间几何计算**

检查线距、环宽、阻焊桥，本质上都是几何碰撞检测问题。走线之间的最短距离、焊盘和钻孔的位置关系，都需要精确的数学计算。

**4. SVG 实时渲染**

解析出来的数据要渲染成可视化的 PCB 图，支持缩放、平移、层切换，最好还能点击违规定位。

说实话，这四项任何一项对产品经理来说都是"不可能完成的任务"。

但 Claude 能做。

## 三、与 Claude 的协作过程

### 第一步：描述我想要什么

我没有写需求文档（虽然我是干这行的），而是直接用大白话告诉 Claude：

> "我要做一个浏览器端的 PCB DFM 分析工具。用户上传 Gerber 文件（ZIP 包），工具自动解析、渲染成 SVG，然后跑 9 项 DFM 检查，最后能生成 PDF 报告。纯前端，零服务器。"

### 第二步：Claude 建议的模块拆分

Claude 很快给出了一个清晰的模块架构：

```
pcb-dfm-tool/
├── index.html                # 单页入口
├── css/style.css             # 深色主题样式
├── js/
│   ├── app.js                # 主应用，编排所有模块
│   ├── gerber/
│   │   ├── parser.js         # Gerber RS-274X 解析器
│   │   ├── layer-id.js       # 层类型自动识别
│   │   ├── renderer.js       # SVG 层渲染
│   │   └── geometry.js       # 几何数据提取
│   ├── dfm/
│   │   ├── engine.js         # DFM 引擎编排器
│   │   ├── rules.js          # 9 项检查规则
│   │   └── spatial-index.js  # R-tree 空间索引
│   ├── ui/
│   │   ├── upload.js         # 拖拽上传 + ZIP 解压
│   │   ├── viewer.js         # SVG 交互查看器
│   │   ├── violations.js     # 违规列表 UI
│   │   └── dashboard.js      # 仪表盘 + 参数面板
│   ├── report/
│   │   ├── pdf-builder.js    # PDF 报告生成
│   │   └── svg-capture.js    # SVG 截图工具
│   └── utils/
│       ├── constants.js      # 常量和厂商预设
│       └── units.js          # mil/mm/inch 转换
```

*（图：初始上传页面 — 打开即用的深色主题界面）*

整个项目 20 个文件，约 5000 行代码，**没有 package.json，没有构建工具，没有 npm install**。

### 第三步：逐模块迭代

Claude 的开发顺序非常合理：

1. **先做解析器**（最难的部分）—— Gerber RS-274X 解析 + Excellon 钻孔解析
2. **再做渲染**—— 解析结果 → SVG 路径，实时渲染到浏览器
3. **然后做检查**—— 9 项 DFM 规则引擎 + 空间索引加速
4. **最后做 UI**—— 上传交互、仪表盘、违规列表、PDF 报告

每一步我都会拿真实的 Gerber 文件去测试，发现问题反馈给 Claude，它来修。整个过程就像和一个非常高效的程序员结对编程。

### 关键数据

| 指标 | 数值 |
|------|------|
| 总代码量 | ~5000 行 |
| 文件数 | 20 个 |
| 服务器成本 | **零** |
| CDN 依赖 | 3 个（JSZip、PDFMake，均从 esm.sh 加载） |
| DFM 检查规则 | 9 项 |
| 支持厂商预设 | 3 个（默认、某制造企业A、某制造企业B） |
| 部署方式 | 静态 HTML，可部署到任意托管平台 |

### 9 项 DFM 检查规则

| # | 规则 | 默认阈值 | 说明 |
|---|------|---------|------|
| 1 | 最小线宽 | 外层 4mil / 内层 3mil | 检查走线宽度是否满足制造要求 |
| 2 | 线距/铜间距 | 6mil | 检查走线之间的最小间距 |
| 3 | 最小钻孔尺寸 | 6mil | 检查过孔和通孔的最小钻孔直径 |
| 4 | 焊盘环宽 | 5mil | 检查焊盘环宽是否足够 |
| 5 | 阻焊桥宽度 | 4mil | 检查相邻焊盘之间的阻焊桥 |
| 6 | 丝印上焊盘 | 不允许重叠 | 检查丝印是否覆盖了焊盘 |
| 7 | 板框完整性 | 必须闭合 | 检查板框轮廓是否完整 |
| 8 | 铜皮到板边距 | 10mil | 检查铜皮到板边的安全距离 |
| 9 | 缺失层检测 | 必须有铜层+板框 | 检查是否缺少必要的 PCB 层 |

所有规则的阈值都可以自定义调节，并且内置了三套厂商预设，一键切换。

### 部署方式

整个工具就是一个 `index.html` + 静态文件。部署方式极简：

```bash
# 方式1：直接浏览器打开
open index.html

# 方式2：本地服务器（推荐）
python3 -m http.server 8080

# 方式3：部署到 Vercel
vercel
```

零服务器成本，零运维负担。
